### 各项目原始定位

| 项目 | 原始定位 | 当前角色 |
|------|---------|---------|
| hermes-desktop | hermes agent 的 Electron 桌面 UI | 不变 — 负责所有 UI 交互 |
| agent-gateway | 多平台消息网关 + CLI agent 桥接 | 仅使用 server/ + agents/ 子模块 |

## 2. 功能模块对比

### 2.1 会话管理 (Session Management)

**冗余程度: 高**

| 层级 | 组件 | 职责 |
|------|------|------|
| 前端 (React) | `$activeSessionId`, `$sessions`, `use-session-actions.ts` | UI 状态管理、会话生命周期控制 |
| Electron 主进程 | `hermes:api` IPC handler, 代理 REST 请求 | HTTP 代理、认证令牌管理 |
| agent-gateway server | `SessionManager` + `DesktopSession` | 内存会话管理、agent 桥接实例化 |
| agent-gateway core | `SessionStore` + `Session` | 平台会话（按 chat/user 分组）、历史裁剪 |

**分析**:
- `core/session.py` 的 `SessionStore` 为 Telegram/Discord 等平台设计，按 `platform:chat_id:user_id` 键值管理会话，与桌面客户端的会话概念完全不同
- `server/session_manager.py` 的 `DesktopSession` 才是桌面需要的——但它只有内存存储，没有持久化
- 前端维护了独立的会话列表状态（`$sessions`），与网关的 `SessionManager` 没有直接同步机制

**建议**: `core/session.py` 在桌面场景下完全冗余，可移除。`server/session_manager.py` 需要补充持久化能力（如 SQLite）。

### 2.2 配置管理 (Configuration)

**冗余程度: 高**

| 层级 | 组件 | 职责 |
|------|------|------|
| 前端 | `hermes.ts` 中 ~10 个 config/env 函数 | REST 调用获取/设置配置 |
| agent-gateway server | `rest_config`, `rest_env` 等 REST 端点 | 返回 stub 响应 |
| agent-gateway server | `handle_config_get/set` (RPC) | 管理 `default_agent_type` |
| agent-gateway core | `GatewayConfig` (YAML) | 平台配置、流式配置、会话配置 |

**分析**:
- `core/config.py` 的 YAML 配置系统（`GatewayConfig`, `PlatformConfig`, `StreamingConfig`）专为多平台网关设计，桌面场景不需要
- 前端调用大量 `/api/config`、`/api/env`、`/api/config/schema` 等端点，当前全部返回空 stub
- 配置管理存在三层：前端 store → REST API → 后端实际存储，中间层（agent-gateway）几乎是透传

**建议**: `core/config.py` 在桌面场景下冗余。agent-gateway 需要实现真实的配置存储（而非 stub），或者将配置管理完全交由 Electron 主进程处理。

### 2.3 模型/提供商管理 (Model & Provider)

**冗余程度: 低**

| 层级 | 组件 | 职责 |
|------|------|------|
| 前端 | `$currentModel`, `use-model-controls.ts`, `model-menu-panel.tsx` | UI 选择器、状态管理 |
| agent-gateway server | `rest_model_options`, `handle_model_options` | 返回可用模型列表 |
| agent-gateway agents | `agent_factory.py` | 根据 agent 类型创建桥接 |

**分析**: 模型管理职责清晰——前端负责 UI，网关负责提供可用模型列表和切换 agent 类型。冗余度低。

### 2.4 消息传递子系统 (Messaging Core)

**冗余程度: 完全冗余**

agent-gateway 的 `core/` 子系统是一个完整的多平台消息处理框架：

| 模块 | 功能 | 桌面是否需要 |
|------|------|-------------|
| `runner.py` (GatewayRunner) | 编排所有平台适配器、消息路由 | 不需要 |
| `adapter.py` (BasePlatformAdapter) | 平台适配器基类（connect/disconnect/send） | 不需要 |
| `message.py` (MessageEvent/MessageSource) | 消息事件数据模型 | 不需要 |
| `delivery.py` (DeliveryRouter) | 消息投递路由 | 不需要 |
| `stream.py` (StreamConsumer) | 平台消息流式渲染 | 不需要 |
| `stream_events.py` | 流式事件类型定义 | 不需要 |
| `registry.py` (PlatformRegistry) | 平台适配器注册表 | 不需要 |
| `session.py` (SessionStore) | 平台会话存储 | 不需要 |

**分析**: 这是 agent-gateway 中最大的冗余模块。`GatewayRunner` 负责从 Telegram/Discord 等平台接收消息、调用 AI agent、将响应投递回原平台。桌面客户端有自己的 UI 层，完全不需要这套消息管道。

### 2.5 平台适配器 (Platform Adapters)

**冗余程度: 完全冗余**

| 适配器 | 功能 | 桌面是否需要 |
|--------|------|-------------|
| `telegram.py` | Telegram Bot API 集成 | 不需要 |
| `discord.py` | Discord Bot 集成 | 不需要 |
| `slack.py` | Slack App 集成 | 不需要 |
| `webhook.py` | Webhook 回调 | 不需要 |

**分析**: 桌面客户端通过 UI 直接与用户交互，不需要任何第三方平台适配器。前端虽有 `/api/messaging/platforms` 相关的设置页面，但当前返回空列表即可。

### 2.6 CLI Agent 桥接 (Agent Bridges)

**冗余程度: 无冗余 — 核心必需**

| 模块 | 功能 | 桌面是否需要 |
|------|------|-------------|
| `base.py` (CLIAgentBridge) | CLI 子进程管理、流式输出 | 核心必需 |
| `claude_code.py` | Claude Code CLI 桥接 | 核心必需 |
| `codex.py` | OpenAI Codex CLI 桥接 | 核心必需 |
| `pi_agent.py` | Pi Agent CLI 桥接 | 核心必需 |
| `agent_factory.py` | 按类型创建桥接实例 | 核心必需 |

**分析**: 这是 agent-gateway 对桌面客户端的核心价值——将本地 CLI 工具（claude、codex、pi）封装为异步流式接口。`SubprocessPool` 管理子进程生命周期，`CLIAgentBridge.stream()` 提供 `AsyncIterator[str]` 流式输出。

## 3. JSON-RPC 方法覆盖率

前端共调用 **22 个** JSON-RPC 方法，agent-gateway 仅实现了 **10 个**：

### 已实现 (10 个)

| 方法 | 处理器 | 用途 |
|------|--------|------|
| `session.create` | `handle_session_create` | 创建新会话 |
| `session.resume` | `handle_session_resume` | 恢复已有会话 |
| `session.close` | `handle_session_close` | 关闭会话 |
| `session.list` | `handle_session_list` | 列出活跃会话 |
| `prompt.submit` | `handle_prompt_submit` | 提交提示并流式返回 |
| `model.options` | `handle_model_options` | 获取模型列表 |
| `commands.catalog` | `handle_commands_catalog` | 获取斜杠命令列表 |
| `config.get` | `handle_config_get` | 获取配置 |
| `config.set` | `handle_config_set` | 设置配置 |
| `tools.list` | `handle_tools_list` | 列出工具集 |

### 未实现 (12 个)

| 方法 | 调用位置 | 优先级 | 说明 |
|------|---------|--------|------|
| `session.interrupt` | `use-prompt-actions.ts` | **高** | 中断当前生成 |
| `session.title` | `use-prompt-actions.ts` | **高** | 生成会话标题 |
| `session.steer` | `use-prompt-actions.ts` | 中 | 追加引导指令 |
| `session.usage` | `use-session-actions.ts` | 中 | 获取会话 token 用量 |
| `slash.exec` | `use-prompt-actions.ts`, `use-model-controls.ts` | **高** | 执行斜杠命令 |
| `command.dispatch` | `use-prompt-actions.ts` | 中 | 分发命令 |
| `setup.status` | `runtime-readiness.ts` | **高** | 启动时检查就绪状态 |
| `setup.runtime_check` | `runtime-readiness.ts` | **高** | 运行时环境检查 |
| `reload.env` | `onboarding.ts` | 中 | 重新加载环境变量 |
| `reload.mcp` | `mcp-settings.tsx` | 低 | 重新加载 MCP 服务器 |
| `complete.path` | `use-context-suggestions.ts` | 低 | 路径自动补全 |
| `image.attach` | `use-prompt-actions.ts` | 低 | 图片附件上传 |
| `image.detach` | `use-composer-actions.ts` | 低 | 移除图片附件 |
| `preview.restart` | `use-preview-routing.ts` | 低 | 重启预览服务器 |

**影响**: 未实现的高优先级方法会导致前端功能异常：
- 缺少 `setup.status` / `setup.runtime_check` → 启动流程可能阻塞
- 缺少 `session.interrupt` → 无法停止正在进行的生成
- 缺少 `slash.exec` → 斜杠命令不可用

## 4. REST API 覆盖率

前端通过 `window.hermesDesktop.api()` 调用 **53+ 个唯一路径**的 REST API。当前 agent-gateway 的 `server/app.py` 为所有路径提供了 stub 实现，但绝大多数返回空/固定数据。

### 按功能分组统计

| 功能组 | 端点数量 | stub 状态 | 实际功能需求 |
|--------|---------|-----------|-------------|
| 会话 (Sessions) | 9 | 全部 stub | 需要真实持久化 |
| 配置 (Config) | 5 | 全部 stub | 需要真实存储 |
| 模型 (Model) | 5 | 全部 stub | 需要连接 agent 配置 |
| 环境变量 (Env) | 6 | 全部 stub | 需要真实 .env 管理 |
| Profile | 8 | 全部 stub | 桌面多 profile 概念需评估 |
| 提供商/OAuth | 7 | 全部 stub | 本地 CLI 不需要 OAuth |
| 技能/工具 | 6 | 全部 stub | 需评估是否需要 |
| 消息平台 | 3 | 全部 stub | 桌面不需要 |
| 定时任务 (Cron) | 7 | 全部 stub | 桌面不需要 |
| 日志 (Logs) | 1 | stub | 可能需要真实日志 |
| 分析 (Analytics) | 1 | stub | 桌面不需要 |
| 音频 (Audio) | 3 | stub | 可能需要 |
| 其他 | 5 | stub | 按需评估 |

## 5. Electron 主进程与 agent-gateway 的职责重叠

### Electron 主进程独有职责（不重叠）

- 后端进程生命周期管理（spawn、kill、restart）
- IPC 通信层（`ipcMain.handle` 注册 ~20 个 handler）
- OAuth 认证窗口管理（Electron session/partition）
- 文件系统访问（配置文件读写、日志）
- 原生通知（`Notification` API）
- 麦克风权限管理
- 远程连接配置管理
- Profile 管理（多桌面 profile）
- 引导安装流程（bootstrap）

### 重叠区域

- **会话持久化**: Electron 主进程通过 `hermes:api` 代理 REST 请求到后端，但 agent-gateway 的 `SessionManager` 只有内存存储，没有数据库。原始 hermes 后端有 `state.db`（SQLite），agent-gateway 缺失这一层
- **配置管理**: 前端通过 REST 调用配置 API，Electron 主进程也管理部分本地配置（`connection-config`），agent-gateway 的配置系统是独立的 YAML 体系

## 6. 冗余总结与建议

### 6.1 可安全移除的代码

以下模块在 hermes-desktop 集成场景下**完全不使用**：

| 路径 | 大小 | 说明 |
|------|------|------|
| `core/runner.py` | ~500 行 | GatewayRunner 消息编排器 |
| `core/adapter.py` | ~640 行 | BasePlatformAdapter 基类 |
| `core/message.py` | ~150 行 | MessageEvent/MessageSource 数据模型 |
| `core/delivery.py` | ~150 行 | DeliveryRouter 投递路由 |
| `core/stream.py` | ~200 行 | StreamConsumer 流式消费器 |
| `core/stream_events.py` | ~100 行 | 流式事件类型 |
| `core/session.py` | ~100 行 | 平台 SessionStore |
| `core/registry.py` | ~80 行 | PlatformRegistry |
| `core/config.py` | ~265 行 | GatewayConfig YAML 配置 |
| `adapters/telegram.py` | ~500+ 行 | Telegram 适配器 |
| `adapters/discord.py` | ~500+ 行 | Discord 适配器 |
| `adapters/slack.py` | ~500+ 行 | Slack 适配器 |
| `adapters/webhook.py` | ~200+ 行 | Webhook 适配器 |

**总计约 3800+ 行代码**，占 agent-gateway 代码量的大部分，但对桌面集成无任何贡献。

**建议方案**: 将 `core/` + `adapters/` 拆分为独立的可选包（如 `agent-gateway[platforms]`），桌面集成仅安装核心 `server/` + `agents/`。

### 6.2 需要补全的功能

**高优先级**（影响基本可用性）:

1. **JSON-RPC 方法**:
   - `setup.status` / `setup.runtime_check` — 启动就绪检查
   - `session.interrupt` — 中断生成
   - `slash.exec` — 斜杠命令执行

2. **会话持久化**: `SessionManager` 需要 SQLite/文件持久化，否则重启后会话丢失

3. **配置存储**: `/api/config` 相关端点需要真实的读写实现

**中优先级**（影响功能完整性）:

4. **环境变量管理**: `/api/env` 需要读写 `.env` 文件
5. **日志查看**: `/api/logs` 需要读取实际的 gateway 日志
6. **模型切换**: `/api/model/set` 需要实际切换 agent 类型

### 6.3 架构调整建议

#### 方案 A: 精简 agent-gateway（推荐）

将 agent-gateway 精简为纯桌面后端：

```
agent-gateway/
├── server/           # FastAPI + WebSocket（保留）
│   ├── app.py        # REST 端点 — 补充真实实现
│   ├── dispatcher.py # JSON-RPC 分发器（保留）
│   ├── methods.py    # RPC 处理器 — 补全缺失方法
│   └── session_manager.py # 补充持久化
├── agents/           # CLI 桥接（保留）
│   ├── base.py
│   ├── claude_code.py
│   ├── codex.py
│   └── pi_agent.py
└── (移除 core/ 和 adapters/)
```

优势：代码量减少 60%+，维护负担大幅降低
劣势：失去多平台消息网关能力（但可作为独立包保留）

#### 方案 B: 保留完整架构，条件加载

保持当前结构，但在 `__main__.py` 入口中仅启动 `server/` 子系统，不加载 `core/runner.py` 和平台适配器。通过 `pyproject.toml` 的 extras 分离依赖：

```toml
[project.optional-dependencies]
desktop = ["fastapi", "uvicorn"]
platforms = ["python-telegram-bot", "discord.py", "slack-sdk"]
```

优势：保留完整功能，按需安装
劣势：代码库仍然庞大，但运行时不加载冗余模块

#### 方案 C: 前端瘦身

hermes-desktop 前端也有大量桌面不需要的功能（messaging 设置页、cron 管理、analytics 仪表盘）。可以考虑：
- 在 UI 中隐藏这些功能模块
- 移除对应的 store 和 hook
- 减少 REST API stub 压力

## 7. 核心结论

1. **agent-gateway 的 `core/` + `adapters/` 子系统**（约 3800 行）在桌面集成场景下**完全冗余**，这是最大的冗余来源

2. **agent-gateway 的 `server/` + `agents/`** 是桌面集成真正需要的部分，但当前实现过于简陋——12 个 JSON-RPC 方法未实现，53 个 REST 端点返回空 stub

3. **会话持久化缺失**是最关键的架构缺陷——原始 hermes 后端有 `state.db`，agent-gateway 只有内存会话

4. **配置体系存在三层混乱**——前端 REST → agent-gateway stub → core YAML，需要统一为单一配置源

5. **平台适配器和消息管道**可以安全拆分为独立可选包，不影响桌面功能
